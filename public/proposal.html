① 문제 정의 — 2/2 05 / 22
주요 불편 사항
⏱
속도 문제
하이퍼캐주얼 게임의 수명이 3~6개월로 단축되며 소재 교체 주기가 주 단위로 빨라졌습니다. 그러나 경쟁사 분석부터 소재 기획까지 수 시간~수일이 걸리는 수작업 구조입니다. 트렌드 대응이 하루만 늦어도 CPI 상승으로 직결됩니다.
✅ Centinel: 검색 → 소재 6종 완성까지 1~3분
📊
일관성 문제
A 담당자와 B 담당자가 동일 검색어로 분석해도 결과가 다릅니다. 개인 역량에 따라 인사이트 품질이 달라지고, 매번 처음부터 수작업으로 시작하는 구조라 일관된 기준이 없습니다. AI 앙상블 구조로 매 분석마다 동일한 분석 프레임과 품질을 보장합니다.
✅ Haiku×5 앙상블 서브에이전트 → Sonnet 선별로 일관 품질
🔄
연결 문제
트렌드 보고서 작성 후 별도로 소재 기획팀에 공유하는 과정에서 맥락이 손실되고, 실행까지 며칠~수주가 소요됩니다. 분석과 생성이 분리되어 있어 트렌드 인사이트가 실제 소재에 반영되는 속도가 느립니다.
✅ 분석 → 광고소재 생성 원스톱 파이프라인
📱
플랫폼 대응
Meta/TikTok 광고는 소재 피로도가 빠르고, 새 소재 없이 같은 소재를 돌리면 CTR 하락 → CPI 상승 → ROAS 악화로 이어집니다. 숏폼 플랫폼은 최소 주 1회 소재 교체가 필요하지만 현재 프로세스로는 공급 속도가 부족합니다.
✅ 훅/메인카피/숏폼대본/앱스토어 설명 즉시 생성
💡 ※ 슈퍼센트 마케터 채용공고(원티드) 및 업계 UA 마케팅 자료 기반으로 작성된 가설입니다. 실제 마케터 인터뷰를 통해 가설 검증 후 고도화 필요.

AI 서브에이전트 패턴 — 역할 분리로 관점 다양성 확보

왜인기인가 Haiku ×1 인사이트 Haiku ×2 광고소재 Haiku ×2 Sonnet 선별
왜 트렌드 + Vision 병렬인가
텍스트(메카닉·키워드)와 시각(UI·훅·색상)을 동시에 분석해야 소재가 구체화됩니다. 병렬 실행이라 추가 시간도 없습니다.
왜 개별 인기 분석인가
전체 트렌드와 별도로, 특정 게임이 왜 지금 차트에 있는지를 해부해야 소재 방향이 구체화됩니다.
왜 앙상블 구조인가
Performance(성과) + Brand(감성) 두 관점 초안 생성 후 최선 선별이 품질 일관성에 유리합니다.

⑤ 프로토타이핑 방법 10 / 22
프로토타이핑 방법
01
데이터 수집 레이어
google-play-scraper로 실제 구글플레이 상위 게임 데이터 수집. genreId.startsWith('GAME') 필터로 비게임 앱 차단.
결과 3개 미만 시 fallback 대신 insufficient_results 에러 반환 + 추천 검색어 칩 표시.
→ genreId 필터 이유: category 파라미터는 search()에서 무시됨. 서버 응답의 genreId로 직접 필터링해야 정확히 게임만 추출.
프로토타입: 결과 3건 미만 시 422 반환. 프로덕션 전환 시 → Supabase 캐시(최근 성공 결과) 또는 자체 크롤링 서버로 대체.
02
Stage 1 — 3-way 병렬 실행
Promise.all 동시 실행 :
Sonnet 트렌드 분석 + Vision(스크린샷 → base64 → Sonnet) + Haiku(왜인기인가 — 차트 게임)
→ Vision 결과를 인사이트·광고소재 앙상블 프롬프트에 주입
추가 지연 없음 — Sonnet이 이미 보틀넥이므로 Vision은 무료 성능. 수동 트리거 제거, 분석 시작 시 자동 실행.
03

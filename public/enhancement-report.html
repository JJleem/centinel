고도화 #2 — API 토큰 최적화 (1/3) 03 / 11
synthesizeAdCopies 선별 방식 개선
왜 변경했나?
기존 코드는 Orchestrator에게 12개 AdCopy 전체를 JSON.stringify(allCopies, null, 2)로 전달했다.

AdCopy 1개에는 8개 필드(hook, mainCopy, shortFormScript, appStoreDescription, targetKeywords, imagePrompt, psychologicalTags, expectedCTR)가 있고, pretty-print 들여쓰기까지 더하면 12개 × 약 280 tokens ≈ 3,400 tokens이 입력 토큰으로 소비된다.
왜 이 방식(Index 선택)을 선택했나?
Orchestrator의 역할은 선별(selection)이지 재생성이 아니다. 어떤 copy가 더 나은지 판단하는 데 필요한 정보는 hook(첫인상)과 expectedCTR(예측 성과) 두 가지면 충분하다.

선택 결과를 index 번호로만 반환받고, 원본 배열에서 그대로 꺼내면 별도 API 호출 없이 full copy를 복원할 수 있다.
기능 변경은 없는가?
△ "Minimal refinement"(미세 수정) 단계 제거 — 기존 프롬프트에 있던 정제 지시를 없애고 원본 그대로 사용
✓ 실질적 품질 차이 없음 — "Minimal refinement only" 지시 자체가 거의 원본을 유지하는 수준이었고, AI 재생성 시 오히려 원본 내용이 변형될 수 있는 리스크가 존재했다
✓ 출력 결과 구조 동일 — 호출부는 여전히 AdCopy[] 6개를 반환받음
Before 8개 필드 × 12개 full copy ≈ 3,400 tokens
JSON.stringify(allCopies, null, 2) // hook · mainCopy · shortFormScript · appStoreDescription // targetKeywords · imagePrompt · psychologicalTags · ctr max_tokens: 4,000 // 6개 full copy 재출력
After hook+ctr만 → index 반환 → 원본 매핑 ≈ 360 tokens
const slim = allCopies.map((c, i) => ({ i, tone: TONE_NAMES[i % 6], hook: c.hook, ctr: c.expectedCTR })); // → {"selected":[0,7,2,9,4,11]} // → TONE_NAMES.map((_, t) => allCopies[selected[t]]) max_tokens: 80 // index 6개만

-89%

고도화 #3 — API 토큰 최적화 (2/3) 04 / 11
visionContext 축약 — 4개 에이전트 공통 비용 절감
왜 변경했나?
Agent V(Vision)가 스크린샷을 분석한 결과를 하나의 문자열로 만들어 4개 에이전트(2A, 2B, 3A, 3B) 프롬프트에 그대로 주입하고 있었다.

각 필드가 2-3문장 full sentence로 구성되어 1회 주입당 약 190 tokens, 4개 에이전트에 동일하게 반복되니 총 760 tokens이 소비된다.
왜 이 방식(첫 문장만 추출)을 선택했나?
에이전트가 visionContext에서 실제로 필요한 정보는 "어떤 느낌의 게임인가"라는 방향성이다. 2-3문장 전체가 아닌 핵심 키 phrase 하나로 충분히 전달된다.

.split(".")[0]으로 첫 문장만, 색상은 앞 2개만, elements는 3개만 추출 — 추가 API 호출 없이 문자열 가공만으로 해결.
기능 변경은 없는가?
△ 세부 묘사 일부 생략 — full sentence의 후반 설명이 잘림
✓ 방향성 전달은 동일 — "탭 기반 조작, 네온 색상, 폭발 이펙트" 같은 핵심 신호는 그대로 전달됨
✓ CLAUDE.md 토큰 룰에서 "향후 최적화 예정" 항목 제거 — 완료된 개선은 문서에서 삭제
Before full sentence × 4 agents ≈ 190 tokens × 4
`\n\nVisual Intelligence (Claude Vision — N screenshots): - Core Interaction: ${visionResult.coreInteraction} - Visual Hook: ${visionResult.visualHook} - Color Palette: ${visionResult.colorPalette} - UI Complexity: ${visionResult.uiComplexity}/5 - Key Visual Elements: ${visionResult.keyVisualElements.join(", ")}`
After key phrase only × 4 agents ≈ 50 tokens × 4
`\n\nVision(${visionResult.analyzedCount} screens): interaction=${coreInteraction.split(".")[0]}; hook=${visualHook.split(".")[0]}; colors=${colorPalette.split(",").slice(0,2).join(",")}; ui=${uiComplexity}/5; elements=${keyVisualElements.slice(0,3).join(", ")}`
절감 효과
Before — 190 tokens × 4회 주입 760 tokens
After — 50 tokens × 4회 주입 200 tokens

-74%

고도화 #4 — API 토큰 최적화 (3/3) 05 / 11
Agent 3A/3B system 프롬프트 공통 블록 상수화
왜 변경했나?
광고 소재를 생성하는 Agent 3A(Performance)와 3B(Brand)의 system 프롬프트를 비교하면 첫 문장(~15 tokens)만 다르고 나머지 전체가 동일하다.

imagePrompt 형식 지정, shortFormScript 조건, psychologicalTags 전체 목록, expectedCTR 기준, tone #6 상세 지침 — 이 5개 블록이 두 함수에 그대로 복붙되어 있어 수정 시 두 곳을 동시에 바꿔야 하는 유지보수 문제와 함께 ~220 tokens 중복이 발생한다.
왜 이 방식(module-level const)을 선택했나?
공통 블록을 AD_COPY_SYSTEM_COMMON 상수로 추출하면 두 함수가 같은 문자열을 참조하게 된다.

토큰 비용 자체는 동일(API는 호출마다 독립적으로 처리)하지만, 프롬프트 수정 시 한 곳만 바꾸면 두 에이전트에 동시 반영된다. 향후 태그 추가·tone #6 수정 시 실수로 두 함수가 달라지는 사고를 원천 차단한다.
기능 변경은 없는가?
✓ API에 전달되는 system 프롬프트 내용 완전히 동일 — 런타임 동작 변화 없음
✓ CLAUDE.md Token rules에서 해당 항목 제거 — 코드에서 명백히 보이므로 문서화 불필요
✓ 이 최적화의 핵심 효과는 토큰 절감보다 유지보수 안전성 — 3A/3B 동기화 실수 방지
Before 두 함수에 동일 블록 복붙
// Agent 3A system: `...performance marketing... imagePrompt: Format: "A [mood]..." Max 60 words. shortFormScript: keep concise, max 60 words. psychologicalTags: "🔥 승부욕 자극", "😌 스트레스..." expectedCTR: 1~10 (one decimal)... For tone #6 (Empathy)...` // Agent 3B — 첫 줄만 다르고 나머지 완전히 동일 system: `...brand storytelling... imagePrompt: Format: "A [mood]..." After 공통 블록을 상수로 추출, 두 함수가 참조

For tone #6 (Empathy)...`; // 3A: 첫 문장만 고유, 나머지는 상수 참조 system: `...performance marketing...\n${AD_COPY_SYSTEM_COMMON}` // 3B: 첫 문장만 고유, 나머지는 상수 참조 system: `...brand storytelling...\n${AD_COPY_SYSTEM_COMMON}`
고도화 #5 — Apple App Store 데이터 통합 06 / 11
Google Play 단독 → iOS 동시 지원
왜 추가했나?
Google Play만 보는 건 시장의 절반만 보는 것이다. 하이퍼캐주얼 게임은 iOS와 Android 유저 성향이 다르고, 플랫폼마다 차트 순위도 다르게 움직인다.

UA 마케터 입장에서 "이 게임이 iOS에서도 뜨고 있는가"는 소재 투자 판단의 핵심 신호다.
왜 하이브리드 방식을 선택했나?
두 플랫폼을 분리 분석하면 AI 파이프라인이 2배로 늘어나 비용과 시간이 2배가 된다. 대신 두 플랫폼 데이터를 합산해 AI 분석은 1회만 실행하되, 차트 표시는 플랫폼별로 구분한다.

같은 게임을 제목 기반으로 매칭해 platform: "both"로 병합 — 비용은 현행 유지, 인사이트만 더 풍부해진다.
추가된 기능
✓ 플랫폼 스위처 — 🤖 Android / 🍎 iOS 전환, 동일한 4탭(글로벌탑·매출탑·캐주얼탑·급상승)
✓ iOS 랭킹 변동 추적 — Supabase에 플랫폼별 스냅샷 저장, rankChange 계산
✓ 양쪽 차트 동시 진입 게임 배지 표시 — Google #5 · iOS #3 형태로 분석 결과에 노출
✓ iOS 게임도 "왜 인기인가?" AI 분석 — iosChartRank 기반으로 analyzeWhyChart 실행
변경된 파일 · 역할
app-store-scraper ← 신규 패키지 types/index.ts platform, iosChartRank, iosRankChange 필드 추가 scrape/route.ts Google + iOS 검색 Promise.all 병렬 실행 제목 매칭 → platform: "both" 병합 charts/route.ts ios-global / ios-grossing / ios-casual 탭 추가 store.app(id) 병렬 조회로 score 보정 cron/snapshot/route.ts iOS 3개 탭 스냅샷 주기 저장 charts/rising/route.ts ?platform=google|ios 파라미터로 급상승 분리 TrendCharts.tsx 플랫폼 스위처 + TAB_URL 매핑 테이블
🔗 제목 기반 매칭 — iOS/Android ID 체계가 달라 직접 매핑 불가. 소문자+특수문자 제거 후 제목 일치로 동일 게임 판별

고도화 #6 — 광고 소재 품질 개선 07 / 11
FOMO 편향 제거 + 피드백 루프
발견된 문제
① FOMO가 항상 1위 — 매 분석마다 4번 FOMO 톤이 expectedCTR 8.6으로 고정 우승. Orchestrator가 CTR 기준으로 선별하는데, AI가 FOMO 톤에 항상 높은 CTR을 부여하는 스테레오타입이 굳어진 것
② CTR이 항상 동일 — "highest 7~9, lowest 4~6" 범위 지시만 있고 변동 지시 없어서 매번 같은 패턴으로 수렴
Fix A — CTR 편향 즉시 제거
✓ Orchestrator slim-select에서 ctr 필드 제거 → hook + mainCopy 품질로만 선별
✓ CTR 지시 재작성 — "FOMO가 항상 높은 게 아님" 명시, 점수 3구간 분산(7.5~9.0 / 5.5~7.4 / 4.0~5.4), "매 분석마다 winner 다르게" 지시
Fix A는 편향을 제거하지만, 어떤 톤이 실제로 좋은지는 여전히 AI 추정에 의존한다.
실제 사용자 선호를 학습하려면 피드백 루프가 필요하다.

고도화 #7 — Vision 분석 결과 캐싱 08 / 11
analyzeVision 반복 호출 제거
왜 변경했나?
analyzeVision은 파이프라인에서 가장 비싼 호출이다. Claude Sonnet + 스크린샷 이미지 base64 인코딩이 조합되어 입력 토큰이 매우 크고, 이미지 다운로드 시간도 추가된다.

같은 검색어를 재분석하거나 비슷한 게임 세트가 반환될 때 동일한 분석을 반복 수행하는 건 비용과 시간 낭비다.
왜 이 캐시 키 방식을 선택했나?
분석 결과는 "어떤 게임의 어떤 스크린샷"에 종속된다. 두 조건이 모두 같아야 동일한 결과를 쓸 수 있다.

appId:screenshotUrl 쌍을 정렬 후 join해 키를 만들면 — 게임 세트가 같으면 항상 같은 키, 스크린샷 URL이 바뀌면 자동으로 키가 달라져 코드 변경 없이 무효화된다.
기능 변경은 없는가?
✓ 분석 결과 동일 — 캐시 HIT 시 기존 Vision 결과를 그대로 반환
✓ 캐시 저장 실패해도 분석 흐름에 영향 없음 (fire-and-forget)
△ 앱 스크린샷이 바뀌어도 URL이 같으면 캐시 재사용 — URL 변경 주기에 따라 최대 수일 지연 가능
캐시 동작 흐름
cacheKey = games .map(g => `${"${g.appId}:${g.screenshots[0]}"}`) .sort().join("|") // DB 조회 const { data: cached } = await supabase .from("vision_cache") .eq("cache_key", cacheKey).single() if (cached) // HIT → Sonnet 호출 없음 return { ...cached.result, analyzedCount } // MISS → 분석 실행 후 저장 supabase.from("vision_cache").upsert({ cache_key: cacheKey, result: parsed })
⚡
재분석 속도
같은 검색어 재분석 시 이미지 다운로드 + Sonnet 호출 전체 스킵. 체감 속도 향상
💰
비용 절감
Vision은 파이프라인 최고가 호출. 캐시 HIT 시 해당 호출 비용 100% 절감
🔄
자동 무효화
앱 업데이트로 스크린샷 URL 변경 시 키가 달라져 자동으로 재분석. 별도 TTL 불필요
🗄️
DB 용량 무시
저장하는 건 이미지가 아닌 텍스트 분석 결과. 게임 1,000개 = 약 1MB

Top 30 → Top 200 & 랭킹 구간 필터
기존 차트는 Top 30만 수집해 상위권 경쟁이 치열한 게임만 보였다. 수집 범위를 200위까지 넓히고, 구간별 필터로 50~200위 틈새 시장을 직접 분석할 수 있도록 했다.
BEFORE
· Cron: num=30 수집
· 차트: Top 30 단일 뷰
· Rising 신규 진입 기준: rank ≤ 30
· surge window: 60분
AFTER
· Cron: Google num=200 / iOS num=100 수집
· 차트: 1-50 / 51-100 / 101-200 / 전체
· Rising 신규 진입 기준: rank ≤ 200
· surge window: 45분 (완화)
DB-first 아키텍처 (iOS 속도 문제 해결)
iOS는 Apple RSS API 한계로 최대 100개만 반환 (num=200 요청해도 100개 응답).
score 개별 조회는 20개씩 배치 처리로 rate limit 방지. Cron이 미리 DB에 저장하고 API는 즉시 응답.
DB 스냅샷이 없거나 2시간 초과 시 live-fetch로 fallback.
